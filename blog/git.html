What are the benefits of version control?

In order to really describe the benefits of Version Control Systems, lets first look at the problems with Manual Version Control.

-Difficult to separate permanent updates from experimental changes. -Very complex if you are dealing with multiple document solutions. -When multiple people are working together, it can be labor intensive to even find the differences between the versions, let alone merge them to a single document.

Version Control (aka Revision Control aka Source Control) lets you track your files over time. Version Control Systems have the following abilities:

-Backup and Restore -Synchronization -Short-term undo -Long-term undo -Track Changes -Track Ownership -Sandboxing -Branching and merging

So Version Control Systems allow multiple people to manage and share files. It enforces discipline by managing the process of passing files to one another. It allows you to archive versions. Maintains historical information to compare versions. Enables collaboration. Recover from accidental deletions or edits. Conserves disc space on control client and on server - uses efficient algorithms to store changes without taking up too much space. Automates the process of keeping revisions. Most support a branching option to keep the master data safe to keep reformatting separate from working version.

To better understand Version Control, we need to look at an actual example. Two of the most popular version control systems are mercurial and git.

How does git help you keep track of changes?

Git is an open-source version control system that allows developers to track changes made on their code files throughout the lifetime of a project. Git was created by Linus Torvalds, the creator of Linux who wanted a better way to track changes in software development. Git helps you to track changes with 3 stages: Committed, modified, and staged.

Committed means that the data is safely stored in your local database. Modified means that you have changed the file but have not committed it to your database yet. Staged means that you have marked a modified file in its current version to go into your next commit snapshot.

To follow the Git workflow do the following -

1. You modify files in your working directory. 2. You stage the files, adding snapshots of them to your staging area. 3. You do a commit, which takes the files as they are in the staging area (index) and stores that snapshotpermanently to your Git directory.

You may be asking yourself, who should use Git? Git is for anyone who wants to track edits to text documents. Great for anyone who needs to share changes with collaborators. Good for anyone not afraid of command line tools. It is not as useful for tracking non-text files, or files that need to be interpreted by a word processor.
